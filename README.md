
# Core Projection Architecture（CPA）
## ― 実存の発見と文脈的射影に基づく遡行的構造設計論 ―

**Version:** 1.0.0  
**Authors:** Aska Lanclaude, neko  
**Date:** 2026-01-01  

---

## Abstract

ソフトウェア構造論においては、長らく二つの問いが独立に扱われてきた。第一に「なぜ構造の中心が存在するのか」という存在論的問い、第二に「なぜ依存は内向きでなければならないのか」という構造制約の問いである。前者は概念モデリングの文脈で語られ、後者は設計規約として提示されてきた。しかし両者を単一の理論的基盤から同時に導出する枠組みは、これまで十分に提示されてこなかった。

本論文は、この理論的空白を埋めるものとして **Core Projection Architecture（CPA）** を提案する。CPAの核心命題は次の通りである。

> *Core は発見されるものであり、発見された Core は文脈的射影として外界に像を結ぶ。依存はこの発現方向の逆をたどる。これが構造制約の根拠である。*

本命題は三つの原理を内含する。第一に、構造の中心（Core）は設計者によって恣意的に構築されるのではなく、業務の本質または実行世界の存在様式として発見される実存である。第二に、発見された Core は特定の目的・要求という文脈によって限定され、その断面が射影像として外界に結像する。第三に、依存関係はこの発現の因果方向を逆行する。像は本体なしに存在し得ないが、本体は像に依存しない。この構造的非対称性を **遡行的依存則（Retro-dependency Principle）** と定義する。

本論文はさらに、実行基盤の存在論的性質が Core の純粋性に与える影響を **存在論的侵食** として定式化し、論理的構成と存在論的構成という二類型を提示する。存在論的侵食はさらに、実行基盤が Core の外側を包む形で影響を及ぼす **外包型侵食（Outside-in Erosion）** と、実行世界の存在様式が Core の定義そのものに混入する **内包型侵食（Inside-out Erosion）** の二形態として分析される。加えて、計算文脈による形式化を通じて、遡行的依存則が型構造として表現可能であることを示す。

CPAは、依存制約を経験則から構造発現の必然へと転換する理論体系である。システムを設計者が恣意的に組み上げる「構築物」としてではなく、発見された実存が外界へと現れてくる「発現プロセス」として捉え直すことで、設計判断の根拠を規約から構造的必然へと再定位する。

## 第1章　序論

### 1.1 問題の所在

ソフトウェア構造設計において、「依存を内向きに保つ」という制約は広く受容されている。外側の構成要素は内側を参照できるが、内側は外側を参照してはならない、という非対称的依存関係である。この原則は変更耐性やテスト容易性といった実践的利点によって正当化されてきた。しかしそれらは構造がもたらす帰結の列挙であり、依存方向そのものの必然性を理論的に導くものではない。規約（コンベンション）として語られるとき、それは「そのように設計すべきである」という規範に留まり、「なぜそうでなければならないのか」という構造的根拠を欠いている。

一方で、構造設計において中心的概念を発見し、それを基盤として体系を構築するという思想もまた広く共有されている。しかし、発見された中心構造がどのように外界へ現れるのか、なぜ中心に置かれなければならないのか、という発現過程の理論は十分に提示されてこなかった。「中心の存在」と「依存の方向」は、それぞれ別個の問題として扱われてきたのである。

この分断は、とりわけ実行基盤が強い存在論的制約を持つ環境において顕在化する。実行環境の物理的・制度的条件が概念構造に影響を及ぼす場合、中心構造の純粋性と実装上の要請が衝突し、理論と実践の乖離が生じる。この現象はしばしば設計の逸脱として理解されるが、実際には構造理論そのものが発現過程を十分に説明していない可能性がある。

本研究は、これらの問題の根底に「構造はいかにして外界へ現れるのか」という発現理論の不在があると仮定する。構造の中心性と依存の方向性は、この発現理論によって初めて単一の命題から導出される。

---

### 1.2 研究目的と理論の三原理

本論文の目的は、構造の中心性と依存制約を単一の命題から導出する理論として CPA を提示することである。

CPA は、システムを設計者が恣意的に組み上げる「構築物」としてではなく、発見された実存が外界へと現れてくる「発現プロセス」として捉え直す。この認識論的転換により、依存制約を設計上のマナーから、構造発現の因果的帰結へと再定義する。

本理論は以下の三原理によって構成される。

**発見原理（Discovery）**  
構造の中心（Core）は設計者が構築するものではなく、業務の本質または実行世界の存在様式として発見されるものである。Core の発見は認識行為であり、Core はその行為以前から潜在的に存在している。

**射影原理（Projection）**  
発見された Core は、それ自体では多次元的な実存であり、外界にそのままの形で現れることはない。特定の目的・要求・制約という文脈という光が当てられることで、特定の断面が射影され、Interface という境界の上に像として結像する。

**遡行原理（Retro-dependency）**  
「実存（Core）→ 射影（Projected Context）→ 現象（Interface）」という発現のプロセスに対し、依存の矢印はその逆方向をたどる。像は本体なしに存在し得ないが、本体は像に依存しない。この構造的非対称性から、依存制約は必然として導出される。

---

### 1.3 本研究の貢献

本論文の主な貢献は以下の通りである。

**(1) 構造中心性の存在論的再定義**  
Core を恣意的構築物ではなく発見対象として位置づけ、構造の中心性に理論的根拠を与える。

**(2) 依存方向の必然化**  
依存制約を発現構造の逆行として導出し、設計規範から構造的必然へと転換する。

**(3) 存在論的侵食の定式化**  
実行基盤の存在論的性質が Core の純粋性に与える影響を分析し、理論の適用範囲を論理的構成・存在論的構成の二類型として整理する。

なお本論文では「論理的構成（Logical Configuration）」と「存在論的構成（Ontological Configuration）」という二項対立を軸とする。
前者はプログラミング言語理論における表示的意味論の拡張として、
実行様式に依存しない抽象的計算構造を指す。
後者は哲学的存在論の転用として、実行基盤がCoreの存在条件そのものを規定する状態を指す。
両者は「抽象的形相」対「具体的存在条件」という対立軸を設計論の文脈で形成する。

なお、「論理的構成」は意味論的構成（Semantic Configuration）とも呼ばれる。
これはプログラミング言語理論における「意味論（Semantics）」——すなわち、
実行様式から独立した計算の意味の記述——に由来する別称であり、両語は同一の概念を指す。
Appendix B の形式化コードにおいては `cpa-semantic` というプロジェクト名がこれに対応する。

**(4) 形式的裏付けの提示**  
計算文脈を用いた抽象化により、遡行的依存則が型構造として表現可能であることを示す。
形式化の詳細は第8章、および Appendix B に収録する。

## 第2章　理論的前提 ― 実存の発見・実行世界の二面性・射影の導入

### 2.1 構造の発見可能性

ソフトウェア設計における根本的な問いは「構造はいかにして確定されるのか」である。
支配的な実践においては、構造とは設計者が要件をもとに組み立てる構築物として扱われてきた。
しかし本論文は、これとは異なる立場を取る。

**構造の中心（Core）は構築されるものではなく、発見されるものである。**

業務ドメインには、設計行為以前から整合的な概念秩序が潜在している。
たとえば「注文には必ず商品と数量と顧客が結びつく」という関係は、それをモデリングする以前から業務の現実として存在している。設計者の役割はこの秩序を生成することではなく、ノイズを取り除き、潜在する構造を言語化することにある。

同様に、特定の実行世界——ゲームエンジン、分散プロトコル、物理制御系——は、その世界が前提とする存在様式をあらかじめ持っている。設計者はその存在様式を自ら創造するのではなく、それを認識し、Coreとして抽出する。

この立場を **Core 発見可能性（Core Discoverability）** と呼ぶ。

Core 発見可能性の帰結として重要な点は、Core の存在が設計者の恣意に依存しないという点である。発見の精度——すなわち、ノイズを排除してどれだけ純粋な核を抽出できるか——は設計者の技量に依存する。しかし発見の対象そのものは、設計者が存在しようとしまいと、そこにある。

---

### 2.2 論理的実存と存在論的実存

Core が発見されるとき、その出自によって性質が二種類に分かれる。
この区別は、後に述べる「存在論的侵食」を理解するための基盤となる。

なお、本節で用いる「論理的（Logical）」とは、
プログラミング言語理論における表示的意味論（Denotational Semantics）の
拡張的用法——すなわち「実行様式から独立した抽象的計算構造」——を指す。
意味（Semantics）は射影という文脈化を経て初めて立ち現れるものであり、
Core そのものは射影以前の純粋な形相（Form）である。
本論文では「論理的実存」をその形相的純粋性を表す語として採用する。

#### 2.2.1 論理的実存（Logical Existence）

純粋な業務ロジック、数学的構造、計算規則に対応する実存である。

論理的実存の本質的な特徴は、実行環境の変更に対して安定であることにある。
たとえば「注文の合計金額は各明細の金額の和である」という規則は、
それがWebサービスとして実装されようと、CLIツールとして実装されようと、
テキストファイルとして記録されようと、変化しない。

この場合、Core は概念的純粋性を維持できる。実行環境は Core の外部に位置し、
Core に構造的影響を与えない。

#### 2.2.2 存在論的実存（Ontological Existence）

実行世界そのものが規定する存在様式（mode of being）に対応する実存である。
空間座標、時間ステップ、物理法則、並行性モデル、ネットワーク越しの合意形成——
これらはある特定の世界が「存在するとはどういうことか」を規定する存在条件であり、
設計者がその世界に存在する限り無視できない。

たとえばUnityのゲームオブジェクトは、その存在様式として三次元空間における
位置・回転・スケールを持つ。これは業務ロジックではなく、その世界の存在論的条件である。
この条件はCoreの定義から切り離すことができない。

この場合、Core は実行世界の存在様式を含んだ形で発見される。
この現象が第5章で定式化する「存在論的侵食」の原因となる。

---

### 2.3 実行基盤の二類型

発見される Core の性質の差異は、それを包む実行基盤の性質と対応している。
実行基盤は構造に対して根本的に異なる二種類の振る舞いをする。

#### 2.3.1 配送装置型（Delivery-type）

Core によって計算された結果を外界へ届けるための媒体である。

WebフレームワークはHTTPという形式でCoreの出力を伝達するが、Coreが何を計算すべきかを決定しない。配送装置型の実行基盤は除去されても Core の意味論的整合性は保持される——ただし外界との接点が失われるだけである。

この類型においては、Core は概念的純粋性を維持できる。実行基盤は Core の外側を覆う層であり、Core の定義に構造的な影響を及ぼさない。

#### 2.3.2 存在基盤型（Foundational-type）

Core がその世界においてのみ存在し得る条件を規定する基盤である。

ゲームエンジンはゲームオブジェクトが存在するための世界の法則そのものを定義する。ブロックチェーンは分散された状態の合意形成という存在様式を規定する。これらの基盤を除去することは、Core の存在条件を消去することに等しい。

この類型においては、Core は実行基盤の存在様式を内包した状態で発見される。

---

### 2.4 射影概念の導入

Core は多次元的な可能性空間である。それ自体は全一的な実存であり、あらゆる文脈に対して潜在的な断面を持つ。しかし外界はつねに特定の目的を持ち、特定の問いを Core に対して投げかける。

「この注文の合計金額を教えよ」「この注文を確定せよ」「未出荷の注文をすべて列挙せよ」——それぞれは Core の異なる断面を要求している。

この操作——特定の文脈（目的・要求・制約）を Core に適用し、その可能性空間から一貫した断面を抽出すること——を **射影（Projection）** と定義する。

射影の操作には本質的な制約がある。

> **射影は Core の可能性空間を限定するのみであり、Core に新たな構造を追加しない。**

これを **射影制約（Projection Constraint）** と呼ぶ。

射影制約は、UseCase や機能単位が独立した存在として設計される傾向への根本的な批判でもある。UseCase は Core の外側に定義された独立層ではなく、Core という実存に文脈という光が当たることで生じる断面——すなわち **Projected Context** である。

Projected Context が Core に存在しない構造を持っている場合、それは射影の誤りであり、Core の発見が不十分であることを示している。

---

### 2.5 構造と実行の分離

本章の締めくくりとして、構造論における重要な直交性を確認する。

**構造（Structure）と実行（Execution）は独立した関心事である。**

構造とは責務の定義と依存関係の体系である。実行とはスレッド・プロセス・サービスといった計算資源の配置方針である。

レイヤの境界と並列化の境界は一致する必要がない。単一のプロセスが複数のレイヤを縦断することは、理論的に何ら問題を生じない。逆に、マイクロサービスに分割されたシステムにおいても、各サービスの内部で同一の構造原理が成立する。

この分離原則は第4章で **実行独立公理** として形式化される。

## 第3章　発現のメカニズム ― Core Projection 仮説

### 3.1 核心命題

第2章で整理した三つの前提——Core 発見可能性・実行世界の二面性・射影概念——は、単一の命題へと収束する。

> **Core は発見されるものであり、発見された Core は文脈的射影として外界に像を結ぶ。依存はこの発現方向の逆をたどる。これが構造制約の根拠である。**

この命題は存在論・発現論・構造制約論の三者を統合する。以下、この命題を構成する三概念を順に定義する。

---

### 3.2 発現プロセスの定式化

Core が外界へと現れるプロセスを **発現（Manifestation）** と呼ぶ。発現は以下の因果系列として定式化される。

```
Core（実存層）
  ↓  [文脈の適用]
Projected Context（射影層）
  ↓  [像の結像]
Interface（現象層）
```

各層の定義は次の通りである。

**Core（実存層）**  
発見された構造の中心。多次元的可能性空間であり、文脈に先行して存在する純粋な実存。

**Projected Context（射影層）**  
特定の目的・要求・制約という文脈によって Core から抽出された断面。Core の部分写像であり、Core を超える構造を持たない。

**Interface（現象層）**  
外界との境界面であり、発現の結像点。副作用の実行——通信・永続化・描画——はここで発生する。Interface は実体を持たず、射影の結果としてのみ存在する像である。

---

### 3.3 Projected Context の構造的位置づけ

Projected Context は独立した論理単位ではない。

これは設計実践における重要な含意を持つ。一般的な設計においては UseCase 層や Application 層が「Core とは別の独立した責務を持つ層」として設計されることが多い。しかし CPA の観点では、そのような「独立した UseCase」の存在は射影概念と矛盾する。

UseCase は Core から文脈的に切り出された断面であり、Core の外部に対して独立した構造を持つことはできない。UseCase が Core に存在しない概念を参照または保持している場合、それは Core の発見が不十分であるか、射影の境界が誤っているかのいずれかである。

---

### 3.4 Interface の役割と副作用の帰属

Interface は発現の終端であり、現象が生起する唯一の層である。

副作用——外部システムへの書き込み、永続層への保存、ユーザへの表示——はすべて Interface において実行される責任を持つ。Core および Projected Context は副作用を**記述**することができる（すなわち、「何を実行すべきか」を型として表現できる）が、その実行責任は Interface に帰属する。

この設計により、Core と Projected Context の純粋性が保証される。純粋性とは「同一の入力に対して常に同一の出力を返す」という性質であり、テスト容易性・推論可能性・再利用性の根拠となる。

---

### 3.5 遡行的依存則（Retro-dependency Principle）

依存の定義を「ある要素の成立に他の要素の存在を必要とすること」とする。

発現の因果系列は次の方向である。

```
Core  →  Projected Context  →  Interface
```

像は本体なしに存在し得ない。Interface は Projected Context を前提としなければ結像できない。Projected Context は Core を前提としなければ断面を持てない。したがって依存の矢印は、この因果を逆向きに遡る。

```
Interface  →  Projected Context  →  Core
```

一方、本体は像がどのように結ばれているかを知る必要がない。Core は自身がどのような Interface として現れているかを参照しない。この非対称性が依存制約の根拠である。

この原理を **遡行的依存則（Retro-dependency Principle）** と定義する。

遡行的依存則の意義は、依存制約を経験則から解放するところにある。「Core は外側の層を参照してはならない」という設計規約は、これまで変更耐性やテスト容易性といった実践的利点によって正当化されてきた。しかしそれらは帰結の列挙にすぎない。遡行的依存則は、依存制約を発現の因果構造から必然として導出する。

---

## 第4章　公理系

第3章の仮説を形式的基盤として確立するため、以下の公理を設定する。これらの公理は互いに独立しており、組み合わせることで CPA の全命題が導出される。

---

### 公理1：Core 発見可能性公理

> 構造の中心（Core）は、設計行為に先行して対象世界に潜在的に存在する。設計者はこれを生成するのではなく発見する。

この公理は、構造の中心性に対して存在論的根拠を与える。Core は設計者の恣意的決定ではなく認識行為の対象であるため、その妥当性は設計者の意図ではなく発見の精度によって判断される。

---

### 公理2：射影制約公理

> 射影は Core の可能性空間を限定する操作であり、Core に新たな構造を追加しない。

この公理は Projected Context の性質を規定する。Projected Context は Core の断面であり、Core を超える構造を保持できない。したがって、独立した UseCase 層が Core に存在しない概念を定義することは、この公理に違反する。

---

### 公理3：遡行依存公理

> 依存は発現方向の逆向きにのみ許容される。Core から Interface への方向に沿う依存は存在しない。

この公理は依存制約の形式的根拠である。「Core は Interface を参照しない」「Projected Context は Interface を参照しない」という制約はこの公理の直接的帰結として導出される。

---

### 公理4：実行独立公理

> 実行単位（Thread / Process / Service）はレイヤ構造の決定要因ではない。構造はレイヤ責務の体系であり、実行は計算資源の配置方針である。両者は独立した関心事として分離される。

この公理は、マイクロサービス化・並行処理・分散化といった実行戦略の選択がレイヤ構造の変更を要求しないことを保証する。

---

### 公理5：フラクタル再帰公理

> Core Projection 構造は任意のスケールにおいて再帰的に成立する。すなわち、Core・Projected Context・Interface という発現の因果系列は、システム全体においても、単一コンポーネントの内部においても、同一の原理として適用可能である。

この公理は CPA の適用範囲を規定する。大規模システム設計から関数レベルの設計に至るまで、同一の構造原理が反復適用できる。

なお、第6章ではこの因果系列に Application Base（制御層）を加えた四層構造として展開する。
Application Base は射影を実行文脈へ降ろす制御機構であり、発現の因果系列に必然的に伴う層として
Projected Context と Interface の間に位置する。公理5が「三層」として記述されているのは
第3章の核心命題（Core → Projected Context → Interface）を継承した略記であり、
実装においては四層（さらに起動制御を担う Boot 層を加えた五層）として具体化される。

---

## 第5章　論理的帰結と存在論的侵食の分析

### 5.1 依存内向き性の必然化

公理3（遡行依存公理）より、依存は発現方向の逆向きにのみ許容される。

発現方向が `Core → Projected Context → Interface` であるから、依存の許容方向は `Interface → Projected Context → Core` に限られる。Core が Projected Context を参照すること、Projected Context が Interface を参照することはいずれも公理3に違反する。

**「依存を内向きに保つ」という制約は、設計規約ではなく公理3の帰結である。**

これは依存制約の論理的地位を根本的に変える。規約として語られるとき、その制約は「それに従うことが推奨される」という規範に留まり、逸脱は設計上の判断として許容の余地を持ちうる。しかし公理的帰結として語られるとき、逸脱は発現構造の破壊として定義される——すなわち、Core が Interface を参照するシステムは、CPA の意味において発現が成立していないシステムである。

---

### 5.2 レイヤ境界と並列境界の非同一性

公理4（実行独立公理）より、レイヤの境界と実行の境界は別個の問題として独立して決定されるべきである。

この帰結は実践において重要な含意を持つ。サービスを分割する（あるいは統合する）という判断は、レイヤ責務の再定義を伴わない。単一のプロセスが Core・Projected Context・Interface の三層を縦断することも、三層を三つのマイクロサービスに分割することも、構造論的には等価である。

誤った設計においては「このサービスは Application Layer だ」「このサービスは Infrastructure Layer だ」という形で実行境界とレイヤ境界が混同される。この混同は、レイヤ責務の曖昧化と依存制約の形骸化を招く。

---

### 5.3 存在論的侵食（Ontological Erosion）の定式化

**定義：** 実行基盤の存在様式が Core の定義に影響を与える現象を **存在論的侵食（Ontological Erosion）** と定義する。

侵食は設計の失敗ではない。それは発見される実存の性質——特に存在論的実存という類型の本質的な帰結——である。存在基盤型の実行環境においては、Core はその世界の存在様式を前提として発見されるほかない。ゲームエンジン上で動作するゲームの Core は、三次元空間という存在論的条件を無視できない。

侵食は二類型に整理される。

#### 5.3.1 外包型侵食（Outside-in Erosion）

配送装置型の実行基盤において生じる侵食である。

実行基盤は Core の外側を覆い、入出力の形式を規定するが、Core 内部の定義には関与しない。
Core が扱う概念は実行基盤の存在様式に依存しない。
侵食の影響は Interface と Application Base に留まり、Core Model の定義は実行世界から独立して成立する。

型レベルでは、実行世界を表す計算文脈がスタックに現れない形として観測される。

```haskell
-- 論理的構成（外包型）の概念的表現
-- 実行世界の計算文脈がスタックに含まれない形を示す
type ProjectedContext a = ReaderT Context (State AppState) a
```

> **注：** 上記は外包型侵食の構造的本質を示す概念的なコード例である。
> 実際の論理的構成では、外部との入出力のために IO を Application Base のスタックに含む実装が通例となる。
> たとえば Appendix B の `cpa-semantic` における実装では
> `ReaderT GlobalConfig (StateT ContextualState IO) a` のように IO を含む形となっており、
> これは「IO が Core の外側を包む（外包型）」という分類の典型例である。
> IO の存在は外包型侵食の証拠ではなく、IO がスタックの**外側**を包む構造であることが外包型の本質である。

#### 5.3.2 内包型侵食（Inside-out Erosion）

存在基盤型の実行環境において生じる侵食である。

実行基盤の存在様式——空間・時間・ライフサイクル・物理法則——は Core の定義そのものに混入する。
Core はその実行世界の中でしか存在できず、世界を除去すると Core の概念も成立しなくなる。
これは設計の失敗ではなく、その実行世界に存在することの必然的帰結である。

型レベルでは、実行世界を表す計算文脈がスタックの内部に現れる形として観測される。

```haskell
-- 存在論的構成（内包型）
-- UnityMonad は Unity 実行世界の計算文脈（内部に IO を含む）
type ProjectedContext a = ReaderT UnityWorld (StateT GameState Unity) a
--                                                              ↑
--                        実行世界の計算文脈がここに現れる = 内包型侵食の型レベル証拠
```

外包型との本質的な差異は、IO（実行世界の計算文脈）がスタックの**外側**に位置するか**内側**に位置するかではなく、
実行世界を表す型（Unity 等）が Core の定義から切り離せない存在条件として組み込まれているかどうかにある。
Appendix B の `cpa-ontological` では `World m` 型クラスとして実行世界を抽象化した実装がこの内包型侵食の型レベル証拠として機能している（Appendix B.2・B.3 参照）。

---

### 5.4 射影原理の適用条件と侵食の関係

侵食の類型は、射影原理の適用条件を変化させる。

外包型侵食においては、射影は Core の純粋な可能性空間を文脈で限定する操作として成立する。射影制約公理は完全な形で適用される。

内包型侵食においては、Core の可能性空間そのものに実行世界の存在論が混入している。射影はその混入を前提とした空間に対して行われる。射影制約公理は依然として成立する——Projected Context は Core を超える構造を追加しない——しかし Core 自体の純粋性が制限されているため、射影の断面もその制限を引き継ぐ。

**侵食は射影原理の無効化ではなく、射影が行われる空間の変容である。**

---

### 5.5 構造安定性条件

以上の分析から、システムが構造的に安定であるための必要十分条件を整理する。

**条件1：Core が正しく発見されている**  
ノイズを排除し、対象世界の潜在的秩序が Core として適切に言語化されている。発見の失敗は、Core に設計者の恣意が混入している状態として現れる。

**条件2：Projected Context が射影制約を満たしている**  
Projected Context が Core の断面として定義され、Core に存在しない構造を独自に保持していない。射影制約の違反は、UseCase 層が独立したドメイン知識を保持する形として現れることが多い。

**条件3：依存が遡行的依存則に従っている**  
依存の矢印が発現方向の逆向きに一貫している。逆向き依存の混入は、Core が Interface の存在を参照すること、あるいは Projected Context が実行の詳細を知ることとして現れる。

いずれかの条件が崩れるとき、構造は不安定化する。条件1の崩れは Core の再発見によって回復する。条件2・3の崩れはリファクタリングによって回復する。しかし条件1が崩れたまま条件2・3を維持しようとすることは、誤った Core の上に精緻な構造を積み上げることであり、最終的にはシステム全体の再設計を要する。

この非対称性は、Core 発見可能性公理が公理系の論理的始点に置かれている理由でもある。

---

## 第6章　レイヤ構成 ― 発現構造の空間的定式化

### 6.1 レイヤ命名の原理

CPA におけるレイヤは、実装慣習や既存アーキテクチャ論の歴史的用語に基づいて定義されない。

既存のアーキテクチャ論が用いてきた語彙——Domain、Application、Infrastructure、UseCase——は、それぞれの文化圏に固有の意味論的な重みを持ち、CPA の構造概念とは独立した含意を帯びている。これらの語彙を流用することは、理論の読み手に既存の設計文化に由来する先入観を持ち込むおそれがある。

CPA のレイヤは **発現の因果系列** に対応して定義される。発現系列とは次の因果連鎖である。

```
Core Model  →  Projected Context  →  Application Base  →  Interface
（根源層）         （射影層）              （制御層）           （現象層）
```

この系列は「実存が現象へ至る経路」であり、レイヤ構成はこの経路の空間的表現である。命名の変更は語の置き換えではなく、概念の再定義である。

---

### 6.2 四層の定義

CPA は以下の四層によってレイヤ構成を定義する。

```
┌──────────────────────────────┐
│          Interface           │  ← 現象層：像が結ばれる外界との境界
├──────────────────────────────┤
│       Application Base       │  ← 制御層：射影を実行文脈に降ろす機構
├──────────────────────────────┤
│      Projected Context       │  ← 射影層：Core の文脈的断面
├──────────────────────────────┤
│         Core Model           │  ← 根源層：発見された実存
└──────────────────────────────┘
```

各層は依存方向に従って積層される。上位層は下位層に依存する。下位層は上位層を知らない。この非対称性は遡行的依存則の空間的表現である。

---

### 6.3 各層の定義

#### 6.3.1 Core Model（根源層）

**定義：** 発見された実存の定式化。

Core Model は「何が存在するか」を記述する層である。型定義・不変条件・概念の関係構造を保持し、システムにとって最も安定した核となる。「何をするか」という手続きの定義は Projected Context に属する。

Core Model は上位層を参照しない。論理的構成においては純粋型として表現される。存在論的構成においては実行世界の存在様式を内包した形で発見されるが、いずれの構成においても、Core Model は自身より上位の層の存在を知らない。

#### 6.3.2 Projected Context（射影層）

**定義：** 文脈によって限定された Core の断面。

Core Model に特定の目的・要求・制約という文脈が適用されることで、その可能性空間から特定の断面が抽出される。この断面が Projected Context である。

Projected Context は Core Model の部分写像であり（射影制約公理）、Core Model に存在しない構造を独立して保持しない。いわゆる UseCase はこの層の外部的呼称に過ぎない。本質は「Core の文脈的断面」であり、独立した業務ロジックの塊ではない。

Projected Context は副作用の記述（Effect）を持ち得るが、その実行は Interface に委ねられる。CPA は Projected Context が純粋であることを要求しない。Projected Context の純粋性は、侵食分析の結果に依存する——論理的構成では純粋な計算として定義でき、存在論的構成では実行世界の計算文脈を含む形になる。いずれの場合においても、Projected Context が Interface を参照しないという依存制約は変わらない。

#### 6.3.3 Application Base（制御層）

**定義：** 射影を実行文脈へ降ろす制御機構。

Application Base は Projected Context によって記述された計算を「実行する」層である。形式的に言えば、Projected Context というモナドスタックを run し、計算結果を実行世界へ降ろす責務を担う。業務的意味の記述は Projected Context に属し、Application Base はその記述を受け取って実行へ繋ぐ制御機構として機能する。

Application Base の主な責務は以下である。

- 計算文脈の実行（run）
- 状態遷移管理・ライフサイクル管理
- トランザクション境界の確定
- ログ・設定・依存性注入

Application Base が担う状態遷移の構造は次式で定式化される。

```
State × Event → (State, Effect)
```

ここで State はシステムの稼働状態（初期化・稼働中・停止等）であり、Effect は副作用の記述である。Effect の実行は Interface に委ねられる。

#### 6.3.4 Interface（現象層）

**定義：** 副作用の実行点であり、外界との境界。

発現の終端であり、Core という実存が「像」として外界に結ばれる結像面である。通信・永続化・描画・外部API連携はすべて Interface において実行される。

Interface は像であり、実体ではない。Interface の実装が変わっても Core Model の定義は変化しない。Interface は Projected Context が記述した Effect を受け取り、具体的な副作用として実行する責任を持つ。Core Model は Interface を知らない。

---

### 6.4 二方向構造

CPA の四層には二つの方向が同時に存在する。この二方向の同時存在が CPA の構造的核心である。

**発現方向（上向き）：** Core から Interface へ向かう方向。意味が具体化し、像が結ばれる方向である。

```
Core Model  →  Projected Context  →  Application Base  →  Interface
```

**依存方向（下向き）：** Interface から Core へ向かう方向。成立条件を遡る方向であり、遡行依存公理の空間的表現である。

```
Interface  →  Application Base  →  Projected Context  →  Core Model
```

上向きの発現が Core の発見から始まる。発見された Core は文脈によって射影され、制御層によって実行に降ろされ、外界に現象として結像する。下向きの依存は、この発現を歪めないための条件である。依存を守ることが発現を守ることである。

---

### 6.5 レイヤと実行単位の独立性

レイヤ境界は責務の境界である。実行単位（Thread / Process / Service）は計算資源の境界である。両者は独立した関心事である（実行独立公理）。

- 単一スレッドが四層を縦断することは問題を生じない
- 各層を別プロセスに分離することも構造論的には等価である
- 並列化・分散化はシステム境界で決定される実行戦略であり、レイヤ責務の変更を要求しない

---

### 6.6 フラクタル再帰性

フラクタル再帰公理（公理5）により、四層構造は任意のスケールで再帰的に成立する。システム全体・サブシステム・単一モジュールのいずれにおいても、同一の構造原理が適用可能である。

ただし再帰的適用が正当化されるのは、その適用が対象スケールにおいて発見された実存の構造に対応している場合のみである。構造のための構造化は発見可能性公理（公理1）への違反である。

---

## 第7章　実行世界の類型と侵食理論

### 7.1 侵食の定義と位置づけ

**定義：** 実行世界の存在様式が Core Model の定義に影響を与える現象を **存在論的侵食（Ontological Erosion）** と呼ぶ。

侵食は設計の失敗ではない。侵食は、発見される実存の性質に由来する必然的帰結である。

侵食分析は設計の起点となる問いである。「この実行世界において、Core はどの種類の実存として発見されるか」という問いへの答えが、以降の設計方針を決定する。

---

### 7.2 侵食分析の判定手順

**第一問：実行基盤を除去した場合、Core Model の概念は成立するか。**

成立する → 実行基盤は配送装置型。**論理的構成** を採用する。

成立しない → 実行基盤は存在基盤型。**存在論的構成** を採用する。

**第二問（存在論的構成の場合）：侵食はどの層まで及ぶか。**

侵食の範囲を識別することで、各層の設計方針が確定する。

---

### 7.3 論理的構成（Logical Configuration）

実行基盤が配送装置型として機能し、Core Model が実行世界の存在様式から独立して成立する構成である。

実行基盤が変更・除去されても Core Model の本質は変化しない。実行基盤の影響は Interface と Application Base において吸収され、Core Model の定義には混入しない。

典型的事例として Webアプリケーション・MCPサーバ・CLIツール・バッチ処理が挙げられる。HTTP プロトコルや JSON-RPC は「配送装置」であり、業務概念の定義に関与しない。通信が消えても「注文」「契約」「計算」という概念は存続する。

---

### 7.4 存在論的構成（Ontological Configuration）

実行基盤が存在基盤型として機能し、Core Model が実行世界の存在様式を内包した形で発見される構成である。

実行基盤を除去することは Core Model の存在条件を消去することに等しい。空間座標・時間ステップ・物理法則・ライフサイクルが Core Model の定義に含まれることは、その世界に存在することの必然的帰結である。

典型的事例としてゲームエンジン（Unity・Unreal）上のシステムが挙げられる。Unity においては `GameObject` が存在単位であり、`Transform` が空間の定義であり、`Time` が時間の定義である。これらは Core Model の外側を「包む」のではなく、Core Model が何者であるかを内側から「定義する」。

---

### 7.5 侵食の二タイプ

#### 外包型侵食（Outside-in Erosion）

配送装置型の実行基盤が生じさせる侵食である。実行基盤の影響は Core Model の外側で吸収される。Core Model の定義は実行世界から独立して成立する。

「外包型」とは、実行基盤が Core Model の外側を包む形で影響を及ぼすことを表している。Core Model は包まれているが、内部に侵入されていない。

#### 内包型侵食（Inside-out Erosion）

存在基盤型の実行環境が生じさせる侵食である。実行世界の存在様式は Core Model の定義そのものに混入し、Core Model は実行基盤を内包した形で発見される。

「内包型」とは、実行基盤の制約が Core Model の内側に含まれる形で影響を及ぼすことを表している。

---

### 7.6 侵食と依存方向の独立性

侵食に関する最も重要な命題を明示する。

> **侵食は依存の方向を変えない。侵食は、依存が行われる空間の性質を変えるだけである。**

存在論的構成において Core Model が実行世界の存在様式を内包していたとしても、依存の方向性は変わらない。Interface が Core Model を参照することは許容されず、Core Model が Interface を参照することは禁止される。

存在論的構成を採用したシステムが「どうせ実行基盤に依存するのだから依存方向など意味がない」という結論に至ることがある。しかし侵食の受容は遡行依存則の放棄ではない。侵食が許容されるのは、その侵食が発見された実存の性質に由来する場合のみであり、設計の便宜による依存方向の逆転を正当化しない。

---

### 7.7 既存アーキテクチャ論との関係

CPA は既存のアーキテクチャ論を否定しない。CPA は、それらの理論が「論理的構成」においてのみ適切に機能していた理由を説明し、「存在論的構成」においてもなお一貫した原理を適用するための拡張理論として位置づけられる。

概念の純粋性を原則として強調する既存のアーキテクチャ論は、論理的構成においては完全に正当である——実行基盤が配送装置型である環境では、Core は純粋に保たれるべきであり、保てる。

しかしこれらの理論は、概念の純粋性が原理的に達成できない存在論的構成に対して、十分な処方を提示してこなかった。「できれば純粋にすべき」という規範は、存在基盤型の環境において設計者を無用に苦しめるか、現実との折り合いをアドホックな妥協として処理するかという二択を迫る。

CPA は侵食を定式化することで、この問題を理論内部で解決可能な問題として再記述する。存在論的構成において Core Model が実行世界の存在様式を内包することは、発見可能性公理の帰結として正当化される。設計上の妥協ではなく、構造的必然として理解される。

---

## 第8章　形式的裏付け ― 計算文脈による証明

### 8.1 形式化の目的

本章では、第3章から第7章で展開した CPA の理論的命題を、計算文脈（Computational Context）の概念を用いて形式化する。

形式化の目的は二つである。第一に、遡行的依存則が型レベルで表現可能であり、型システムによって機械的に強制されることを示すこと。第二に、存在論的侵食が型シグネチャの変化として観測可能であることを示すことである。

形式化言語として Haskell を用いる。Haskell の型システムは計算文脈を型として明示的に追跡する性質を持つため、CPA の命題を直接的に表現するのに適している。ただし CPA の命題は Haskell 固有のものではなく、計算文脈を型として表現できる任意の言語において同等の表現が可能である。

---

### 8.2 状態遷移の定式化

Application Base が担う状態遷移は次の形式で定式化される。

```
State × Event → (State, Effect)
```

Effect は副作用の記述であり、実行ではない。この分離が「何をすべきか」の論理と「どのように実行するか」の実装を切り離す。

Haskell ではこの構造を次のように表現する。

```haskell
-- Application Base の状態と遷移イベント
data AppState = Initializing | Running | Suspended | Terminating

data Event
  = StartEvent Config
  | SuspendEvent
  | ResumeEvent
  | TerminateEvent

-- Effect：副作用の記述（実行ではない）
data Effect
  = NoEffect
  | LogEffect Text
  | NotifyEffect Text
  | ShutdownEffect

-- 状態遷移関数：純粋関数として定義される
transition :: AppState -> Event -> (AppState, Effect)
transition Initializing (StartEvent _)  = (Running,     LogEffect "System started")
transition Running      SuspendEvent    = (Suspended,   LogEffect "System suspended")
transition Suspended    ResumeEvent     = (Running,     LogEffect "System resumed")
transition Running      TerminateEvent  = (Terminating, ShutdownEffect)
transition state        _               = (state,       NoEffect)
```

`transition` は IO を持たない純粋関数である。Effect の実行は Interface が担う。この構造により、状態遷移ロジックは実行環境を必要とせずにテスト・検証できる。

---

### 8.3 計算文脈としての実行世界

実行基盤を計算文脈として定式化する。

計算文脈とは「ある計算がどのような前提・環境・状態の下で実行されるか」を型として記述したものである。実行基盤は以下の三つの性質として分解できる。

- **環境（Reader 的性質）：** 実行を通じて変化しない読み取り専用の情報。設定値・実行世界の定数など。
- **状態（State 的性質）：** 実行を通じて変化しうる可変情報。業務状態・ゲーム状態など。
- **計算文脈そのもの：** その世界においてのみ意味を持つ実行文脈。論理的構成では外部の副作用として現れ、存在論的構成では計算スタックの内部に組み込まれる。

---

### 8.4 論理的構成における Projected Context の型表現

論理的構成において、Projected Context は実行世界の計算文脈をスタックに含まない。

```haskell
-- 論理的構成：実行世界の計算文脈がスタックに含まれない
type ProjectedContext a = ReaderT Context (State AppState) a
```

この型は「Context という環境を読み取り、AppState を変更し、結果を返す計算」を表す。実行世界の計算文脈はスタックに現れない。

具体例として、アバター発話ユースケースを示す。

```haskell
-- Core Model（実行世界に依存しない純粋な型定義）
data Avatar      = Avatar { avatarName :: Text, avatarState :: AvatarState }
data AvatarState = Idle | Walking | Speaking
data AvatarEvent = StartWalk | StopWalk | SpeakEvent Text

-- Core Model の純粋関数
updateAvatarState :: AvatarState -> AvatarEvent -> AvatarState
updateAvatarState Idle  StartWalk      = Walking
updateAvatarState _     StopWalk       = Idle
updateAvatarState _    (SpeakEvent _)  = Speaking
updateAvatarState state _              = state

-- Projected Context（"アバターに発話させる" という文脈の射影）
data SpeakRequest = SpeakRequest { message :: Text }
type AvatarStore  = Map Text Avatar

usecaseSpeak :: ReaderT SpeakRequest (State AvatarStore) AvatarState
usecaseSpeak = do
  req          <- ask
  currentState <- gets (avatarState . fromJust . Map.lookup "player")
  let newState  = updateAvatarState currentState (SpeakEvent (message req))
  modify $ Map.adjust (\a -> a { avatarState = newState }) "player"
  return newState
```

型シグネチャ `ReaderT SpeakRequest (State AvatarStore) AvatarState` は「SpeakRequest という文脈を受け取り、AvatarStore を変更し、AvatarState を返す」という Projected Context の意味を、ドキュメントなしに表現している。型シグネチャがアーキテクチャの記録として機能する。

Application Base はこの Projected Context を `run` して実行世界へ降ろす。

```haskell
-- Application Base：Projected Context を run して実行世界へ降ろす
runUsecaseSpeak :: SpeakRequest -> AvatarStore -> IO (AvatarState, AvatarStore)
runUsecaseSpeak req store =
  return $ runState (runReaderT usecaseSpeak req) store
```

`runReaderT` と `runState` による `run` がここで行われる。Projected Context は計算の記述として存在し、Application Base がその記述を実行へ降ろす。

---

### 8.5 存在論的侵食の型シグネチャへの観測

存在論的構成においては、実行世界の計算文脈が Projected Context のスタックに現れる。これが侵食の型レベル証拠である。

```haskell
-- ■ 論理的構成（外包型）
-- 実行世界の計算文脈がスタックに含まれない
type ProjectedContext a = ReaderT Context (State AppState) a

-- ■ 存在論的構成（内包型：Unity 等）
-- Unity という実行世界の計算文脈がスタックの内部に組み込まれる
-- = 内包型侵食の型レベル表現
type ProjectedContext a = ReaderT UnityWorld (StateT GameState Unity) a
--                                                              ↑
--                        実行世界の計算文脈がここに現れる = 侵食の証拠
```

`Unity` は Unity 実行世界を表す計算文脈の型である。その内部に IO を含む構造を持つが、CPA の観点では「Unity という実行世界の計算文脈」として扱う。論理的構成との対比において重要なのは、IO の有無ではなく、**実行世界を表す計算文脈がスタックに内包されているかどうか**である。

この型シグネチャの差異は、発見された Core がどの種類の実存であるかを形式的に記録している。

#### 存在論的構成の具体例

Unity のゲームシステムにおいて、プレイヤーの移動は Unity 実行世界を前提とせずには記述できない。

```haskell
-- 存在論的構成における Core Model
-- Transform は Unity 世界の存在様式であり、Core の定義に混入している
data PlayerCore = PlayerCore
  { playerId      :: PlayerId
  , transform     :: Transform   -- Unity の存在様式を内包
  , health        :: Health
  , movementSpeed :: Float
  }

-- 存在論的構成における Projected Context
-- Unity という実行世界の計算文脈がスタックに組み込まれている
type UnityContext a = ReaderT UnityWorld (StateT GameState Unity) a

usecaseMovePlayer :: Vector3 -> UnityContext ()
usecaseMovePlayer direction = do
  player <- gets currentPlayer
  let newPos = calculatePosition (transform player) direction (movementSpeed player)
  liftUnity $ setTransformPosition (transform player) newPos  -- Unity API 呼び出し
  modify $ updatePlayerTransform newPos
```

`liftUnity` が使えるのは `UnityContext` が Unity 計算文脈をスタックに含んでいるからである。`setTransformPosition` への依存は設計上の妥協ではなく、Unity という世界においてプレイヤーが存在するための必然的帰結として正当化される。

---

### 8.6 遡行依存則の型レベル証明

遡行的依存則は、型システムによって機械的に強制される。

計算文脈のスタックにおいては、上位文脈が下位文脈を `run` することは可能である。しかし下位文脈から上位文脈を `run` することは型システムが禁止する。

```haskell
-- ✅ 許容される方向：Application Base が Projected Context を run する
runUsecaseSpeak :: SpeakRequest -> AvatarStore -> IO (AvatarState, AvatarStore)
runUsecaseSpeak req store =
  return $ runState (runReaderT usecaseSpeak req) store

-- ❌ 禁止される方向：Projected Context が上位文脈を run しようとする
illegalUsecase :: ReaderT SpeakRequest (State AvatarStore) ()
illegalUsecase = do
  -- ReaderT + State のスタックは上位の IO 文脈を run できない
  -- → 型エラー（コンパイル時に検出される）
  runIO $ putStrLn "This violates Retro-dependency Principle"
```

存在論的構成においても同じ原理が成立する。

```haskell
-- ✅ 許容される方向：Interface が Projected Context を run する
runMovement :: Vector3 -> GameState -> UnityWorld -> Unity GameState
runMovement dir state world =
  execStateT (runReaderT (usecaseMovePlayer dir) world) state

-- ❌ 禁止される方向：Projected Context が Interface を直接呼び出す
-- usecaseMovePlayer の内部は、どの Interface がこれを run するかを知らない
```

この型制約が示す帰結は次の通りである。

> **遡行的依存則は、適切に型付けされたシステムにおいてはランタイムエラーではなくコンパイルエラーとして現れる。**

依存制約は設計者のマナーではなく、型の構造的必然である。

---

### 8.7 形式化の帰結と限界

**帰結1：依存方向の機械的強制**  
遡行的依存則に反するコードは、型レベルでコンパイルエラーとして現れる。依存制約は規約から構造的必然へと転換される。

**帰結2：侵食の型シグネチャによる観測**  
存在論的侵食は Projected Context の型シグネチャへの実行世界の計算文脈の混入として観測される。型シグネチャはアーキテクチャ選択の形式的記録として機能する。

**帰結3：lift / run の非対称性による証明**  
上位文脈が下位文脈を `run` することは許容されるが、逆は型システムが禁止する。この非対称性が遡行依存公理の形式的証明である。

一方、本形式化には以下の限界がある。

**限界1：侵食範囲の定量化**  
型シグネチャが侵食の有無を示すことはできるが、侵食がどの範囲まで許容されるかの境界は定性的判断に依存する。侵食判定の定量化は今後の課題として残る。

**限界2：Core 発見精度の非検証性**  
型が正しく付いていても、型を与えた Core の概念が誤って発見されている場合、型システムはその誤りを検出できない。型整合性は構造安定性の必要条件であるが十分条件ではない。

**限界3：言語依存性**  
本形式化は Haskell の型システムを媒体とする。静的型付けを持たない言語、あるいはモナドトランスフォーマを直接サポートしない言語においては、同等の保証を別の手段で実現する必要がある。C# の `Task<T>`・Rust の `Result<T, E>`・TypeScript の型パラメータなど、各言語固有の計算文脈表現が CPA の命題を示す媒体となり得る。

これらの限界は、第9章の評価において正面から議論する。

## 第9章　評価と実践的含意

### 9.1 評価枠組みの設定

本章では、CPA を理論的・実践的観点から総合的に評価する。評価は以下の七軸に基づいて行う。

1. **理論的一貫性**
2. **適用範囲**
3. **実装指針の明確さ**
4. **テスト容易性**
5. **概念的独自性**
6. **説明可能性**
7. **スケーラビリティ**

これらの評価軸は、CPA が単なる設計指針ではなく、存在論的基盤を持つ構造理論として妥当であるかを検証するために設定される。

---

### 9.2 理論的一貫性

CPA は五つの公理を起点として、依存制約・層構造・侵食概念を演繹的に導出する体系である。

依存の内向き性は遡行依存公理から直接導かれる。Projected Context が Core の部分写像であることは射影制約公理の帰結である。レイヤ境界と実行境界の非同一性は実行独立公理から導かれる。侵食の二類型は Core 発見可能性公理と射影制約公理の結合によって説明される。

各命題は外部理論の援用によらず、公理系内部の論理的帰結として位置づけられている。この意味で CPA は規範的アーキテクチャ論ではなく、演繹的構造理論としての一貫性を保持している。

さらに、理論的一貫性は形式化によって裏付けられる。遡行依存則は型システムにおける `lift` / `run` の非対称性として機械的に強制される。侵食は Projected Context の型シグネチャへの実行世界の計算文脈の混入として観測される。概念レベルの命題と形式レベルの表現が対応していることが、理論的一貫性をさらに強固にしている。

一方で、侵食判定の基準が定性的問いに依存している点は、公理系の完全な形式化という観点からの残課題として正直に認識されなければならない（9.9節参照）。

---

### 9.3 適用範囲

CPA の適用範囲における最も重要な特性は、論理的構成と存在論的構成の双方を理論内部で扱える点にある。

概念の純粋性を原則として強調する既存のアーキテクチャ論は、実行基盤が配送装置型として機能する論理的構成においては有効に機能する。しかし実行基盤が世界の存在様式そのものを規定する存在論的構成においては、純粋性の要請が実践との乖離を生じさせてきた。

CPA は侵食を理論内概念として定式化することで、この乖離を排除の対象ではなく分析の対象へと転換する。

- 実行基盤が除去可能である場合 → 論理的構成
- 実行基盤が存在条件を規定する場合 → 存在論的構成

この区別は価値判断ではなく、発見される存在条件の差異である。侵食の受容は妥協ではなく、その世界に Core が存在するための構造的必然として理解される。

また、フラクタル再帰公理により、同一原理は任意の粒度で適用可能である。システム全体・サブシステム・単一モジュールのいずれにおいても同一構造が成立する（Appendix B.5 参照）。

一方、Core の明瞭さが確認できない領域では適用の恩恵は限定的となる。単純なデータ変換パイプラインのように実存的中心が不在の系では射影構造は不要であり、無理な適用はかえって複雑化を招く。フラクタル再帰は複雑化の手段ではないという原則は、適用限界の判断においても有効な基準となる。

---

### 9.4 実装指針の明確さ

CPA は静的な構造理論であると同時に、設計プロセスへの接続を持つ。

#### 9.4.1 設計の順序

CPA における設計は以下の順序で進む。

**第一段階：実行世界の同定**  
実行基盤が配送装置型か存在基盤型かを判定し、論理的構成と存在論的構成のいずれを採用するかを決定する。

**第二段階：Core の発見**  
脈非依存な最小論理単位を発見する。「この概念は実行世界を除去しても成立するか」という問いが発見の基準となる。

**第三段階：射影の定義**  
要求に応じた断面を Projected Context として構成する。射影は Core に新たな構造を追加しない——Projected Context が Core に存在しない概念を保持している場合、それは射影の誤りか Core の発見不足を示している。

この手順により、設計は構築ではなく発見の精度向上として再定義される。

#### 9.4.2 判定基準の提示

CPA は設計者に具体的な判定基準を提供する。

**Core 発見優先原則**　設計の開始点は要求の列挙ではない。まず「それは文脈を除去しても成立するか」を問い、Core を同定する。設計の質は生成能力ではなく発見精度によって規定される。

**射影限定原則**　Projected Context は Core を拡張してはならない。射影は可能性の限定であり、新たな概念を導入している場合、それは射影の失敗か Core の発見不足を示唆する。

**遡行依存の遵守**　依存は常に現象から実存への遡行的経路をたどる。Core が Interface を参照していること、Projected Context が実行の詳細を知っていることは、いずれも構造的逸脱として定義される。リファクタリングとは、依存の矢印を発現の逆方向に再整列する行為である。

---

### 9.5 テスト容易性

CPA の構造は、テスト容易性を設計原理として担保する。

Application Base の状態遷移は次式として定式化される。

```
State × Event → (State, Effect)
```

Effect は副作用の記述であり実行ではない。この分離により、状態遷移ロジックは実行環境を必要とせずに純粋関数として検証できる。副作用境界が Interface に限定されることで、テストの対象と副作用の実行が明確に切り離される。

論理的構成における Projected Context は純粋な計算として定義されるため、実行環境のモックや依存性注入を必要とせずにテストできる。存在論的構成においては実行世界の計算文脈がスタックに混入するためテストの難度は上昇するが、それは理論の欠陥ではなく実行世界の存在条件の反映である。

---

### 9.6 概念的独自性

CPA の理論的独自性は三点に整理される。

**依存制約の必然化**  
「内向き依存を保つ」という原則を実践的利点の列挙から切り離し、発現の因果構造——像は本体なしに存在し得ない——という単一の命題から導出した。依存制約は規約から構造的必然へと転換される。型システムによる機械的強制は、この必然性の形式的実現である。

**Core の発見的再定義**  
Core を設計者が構築するものではなく発見するものとして位置づけたことで、設計行為の意味が根本的に変わる。設計の質は生成能力ではなく発見精度に依存する。この転換は設計行為の認識論的再定義である。

**存在論的侵食の理論化**  
実行基盤の影響を設計の失敗ではなく発見される実存の性質として再解釈した点は、構造理論における認識論的転回である。既存の理論が対処できていなかった存在基盤型環境への適用を可能にし、理論の適用範囲を実践の現実に合わせて拡張した。

---

### 9.7 説明可能性

CPA は設計判断の根拠を言語化するための語彙を提供する。

「なぜこの概念は Core Model に置かれるのか」「なぜこの処理は Projected Context に属するのか」「なぜ依存はこの方向でなければならないのか」——これらの問いに対して、CPA は発現の因果構造という単一の論拠から一貫した答えを与える。

とりわけ「侵食」という概念は、存在論的構成において設計者が感じる「実行基盤への依存はやむを得ない」という実感を理論として言語化したものである。「これは内包型侵食として受容すべき範囲か」「これは発見の誤りによる不必要な依存か」という議論が、理論に基づいて行えるようになる。

---

### 9.8 スケーラビリティ

フラクタル再帰公理により、CPA の四層構造は任意のスケールで成立する。単一コンポーネントからサブシステム、システム全体に至るまで、同一の構造原理が再帰的に適用可能である。

実行独立公理により、並列化・分散化といった実行戦略の変更はレイヤ構造の変更を要求しない。単一プロセスから分散システムへの移行は、CPA の観点では実行戦略の変更であり、構造原理の変更ではない。

---

### 9.9 残課題：侵食判定の定量化

CPA における主要な残課題は、侵食判定の定量的基準の確立である。

現状、侵食分析は「実行基盤を除去しても Core の概念は成立するか」という定性的問いに依存する。この問いは概念的には明確だが、部分的侵食——Core の一部の概念は実行世界に依存し、別の一部は独立している——という状況において、侵食の範囲を客観的に識別する基準が未整備である。

Appendix B の形式化実装を通じて、侵食判定の初期的な型レベル指標が得られている。
具体的には、Projected Context の型シグネチャにおいて実行世界を表す計算文脈型（`World m` 等）が
スタックの内部に現れるか否かが、外包型と内包型を区別する観測可能な指標として機能することが確認された（Appendix B.1・B.2 参照）。
この観測指標は定量化の起点として有望であるが、部分的侵食の範囲を精度よく識別するには
さらなる指標の整備が必要である。

今後の課題として、以下が考えられる。

- 型シグネチャによる侵食検出指標の体系化（上記観測指標を含む）
- 侵食判定チェックリストの整備
- 侵食範囲のスコアリング手法の開発
- 侵食の受容が設計コスト——変更耐性・移植性・テスト負荷——に与える影響の定量分析

定量化が進むことで、CPA は理論から工学的ツールへと進化し、設計者間のコミュニケーション精度をさらに高めるものと考えられる。

## 第10章　結論 ― 規約から必然への転換

### 10.1 核心命題の再確認

本論文が提示した CPA の核心命題は次の通りである。

> **Core は発見されるものであり、発見された Core は文脈的射影として外界に像を結ぶ。依存はこの発現方向の逆をたどる。これが構造制約の根拠である。**

この命題は、ソフトウェア構造論において長らく独立に扱われてきた二つの問い——「なぜ構造の中心が存在するのか」と「なぜ依存は内向きでなければならないのか」——を、単一の論拠から同時に導出する。

Core が発見される実存である以上、Core は発現の起点として構造の中心に位置する必然がある。発現した像が本体なしに存在し得ない以上、依存は発現の逆をたどらなければならない。この二命題は独立した規約ではなく、発現という単一の因果構造から導かれる帰結である。

---

### 10.2 三原理の統合的意義

CPA は発見・射影・遡行という三原理によって構成される。三原理は独立した設計指針ではなく、発現という単一構造から分岐して展開される必然の系列として理解される。

**発見原理（Discovery）** は、Core の存在論的地位を定める。設計は生成ではなく認識である。Core の妥当性は設計者の意図ではなく発見の精度によって判断される。この転換により、設計行為の評価基準が生成能力から発見精度へと根本的に移行する。

**射影原理（Projection）** は、Core がいかにして外界へ現れるかを定める。Core という多次元的実存に文脈という光が当たることで、特定の断面が抽出される。射影は Core に新たな構造を追加しない。UseCase は Core の外部に独立した実体として設計されるものではなく、Core の文脈的断面として発見されるものである。

**遡行原理（Retro-dependency）** は、発現の因果構造から依存制約を導出する。像は本体なしに存在し得ない。本体は像を知らない。この非対称性から「内向き依存を保つ」という制約が必然として導かれる。規約として語られてきた原則は、ここで初めて因果的根拠を得る。

三原理の統合的意義は、設計理論における「規約から必然へ」という転換にある。設計判断の根拠が「そのようにすべきとされている」という規範から、「そうでなければ発現が成立しない」という因果へと移行する。

---

### 10.3 設計理論における転回

CPA がもたらす最大の変化は認識論的なものである。

システムは設計者が恣意的に組み上げる「構築物」ではなく、発見された実存が外界へと現れてくる「発現プロセス」として捉え直される。設計者の役割は、Core を創造することではなく、潜在する秩序を発見し、その発現を歪めないように構造を整えることである。

この転回は、存在論的侵食の扱いにおいて最も鮮明に現れる。実行世界の存在様式が Core の定義に混入することを「汚染」として忌避するのではなく、その世界に Core が存在するための必然的帰結として受容する。この存在論的誠実さが、理論と実践の乖離を解消する。

ソフトウェアは、論理が現実の制約——時間・資源・空間——の上で結像する現象である。CPA はその像の背後にある実存を重んじ、自然な因果の流れに沿って構造を整えることを求める。

論理的純粋性は、存在条件が課されない抽象空間においてのみ完全に成立する。
現実の実行世界において Core が実存するためには、その世界の存在論的条件に従わなければならない。
存在論的侵食はその必然の現れであり、純粋性の失敗ではなく、世界への誠実な応答である。
設計者に求められるのは侵食を排除することではなく、侵食の範囲を正確に認識し、
論理的構成と存在論的構成のいずれにあるかを自覚した上で構造を整えることである。

---

### 10.4 今後の課題

**侵食判定の定量化**  
9.9節で述べた通り、侵食判定の定性的問いを定量的基準へと展開することが第一の課題である。判定チェックリストの整備・スコアリング手法の開発・設計コストとの相関分析が、理論の工学的展開として求められる。

**分散環境における再帰的適用の精緻化**  
実行独立公理によりレイヤ境界と実行境界の非同一性は保証されているが、大規模な分散システムにおいてサービス境界をどのように決定するかの指針は本論文では扱っていない。CPA の四層構造が複数のサービスにまたがる場合の構造論的整理が課題として残る。

**型システムを持たない環境での保証手法**  
本論文の形式化は静的型システムを前提としている。型システムを持たない言語環境において遡行依存則を保証するための代替的な手法の探索が求められる。

**Core 抽出の支援技術**  
侵食判定の定量化と連動する形で、静的解析や形式検証を用いた Core 抽出の支援技術の開発が、理論の実践的普及に向けた長期的課題として挙げられる。

---

### 終わりに

発現を守ることが依存を守ることであり、依存を守ることが発現を守ることである。

この同一命題の二面性の中に、Core Projection Architecture の理論的核心がある。

かつて「モナドは単なる自己関手の圏におけるモノイド対象だよ」と言われた。
難解に見える構造も、正しい視点に立てば、それ以外にあり得ない必然として現れる。
Core Projection Architecture が示したかったのも、それと同じことである。
