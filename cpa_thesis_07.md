## 第7章　実行世界の類型と侵食理論

### 7.1 侵食の定義と位置づけ

**定義：** 実行世界の存在様式が Core Model の定義に影響を与える現象を **存在論的侵食（Ontological Erosion）** と呼ぶ。

侵食は設計の失敗ではない。侵食は、発見される実存の性質に由来する必然的帰結である。

侵食分析は設計の起点となる問いである。「この実行世界において、Core はどの種類の実存として発見されるか」という問いへの答えが、以降の設計方針を決定する。

---

### 7.2 侵食分析の判定手順

**第一問：実行基盤を除去した場合、Core Model の概念は成立するか。**

成立する → 実行基盤は配送装置型。**論理的構成** を採用する。

成立しない → 実行基盤は存在基盤型。**存在論的構成** を採用する。

**第二問（存在論的構成の場合）：侵食はどの層まで及ぶか。**

侵食の範囲を識別することで、各層の設計方針が確定する。

---

### 7.3 論理的構成（Logical Configuration）

実行基盤が配送装置型として機能し、Core Model が実行世界の存在様式から独立して成立する構成である。

実行基盤が変更・除去されても Core Model の本質は変化しない。実行基盤の影響は Interface と Application Base において吸収され、Core Model の定義には混入しない。

典型的事例として Webアプリケーション・MCPサーバ・CLIツール・バッチ処理が挙げられる。HTTP プロトコルや JSON-RPC は「配送装置」であり、業務概念の定義に関与しない。通信が消えても「注文」「契約」「計算」という概念は存続する。

---

### 7.4 存在論的構成（Ontological Configuration）

実行基盤が存在基盤型として機能し、Core Model が実行世界の存在様式を内包した形で発見される構成である。

実行基盤を除去することは Core Model の存在条件を消去することに等しい。空間座標・時間ステップ・物理法則・ライフサイクルが Core Model の定義に含まれることは、その世界に存在することの必然的帰結である。

典型的事例としてゲームエンジン（Unity・Unreal）上のシステムが挙げられる。Unity においては `GameObject` が存在単位であり、`Transform` が空間の定義であり、`Time` が時間の定義である。これらは Core Model の外側を「包む」のではなく、Core Model が何者であるかを内側から「定義する」。

---

### 7.5 侵食の二タイプ

#### 外包型侵食（Outside-in Erosion）

配送装置型の実行基盤が生じさせる侵食である。実行基盤の影響は Core Model の外側で吸収される。Core Model の定義は実行世界から独立して成立する。

「外包型」とは、実行基盤が Core Model の外側を包む形で影響を及ぼすことを表している。Core Model は包まれているが、内部に侵入されていない。

#### 内包型侵食（Inside-out Erosion）

存在基盤型の実行環境が生じさせる侵食である。実行世界の存在様式は Core Model の定義そのものに混入し、Core Model は実行基盤を内包した形で発見される。

「内包型」とは、実行基盤の制約が Core Model の内側に含まれる形で影響を及ぼすことを表している。

---

### 7.6 侵食と依存方向の独立性

侵食に関する最も重要な命題を明示する。

> **侵食は依存の方向を変えない。侵食は、依存が行われる空間の性質を変えるだけである。**

存在論的構成において Core Model が実行世界の存在様式を内包していたとしても、依存の方向性は変わらない。Interface が Core Model を参照することは許容されず、Core Model が Interface を参照することは禁止される。

存在論的構成を採用したシステムが「どうせ実行基盤に依存するのだから依存方向など意味がない」という結論に至ることがある。しかし侵食の受容は遡行依存則の放棄ではない。侵食が許容されるのは、その侵食が発見された実存の性質に由来する場合のみであり、設計の便宜による依存方向の逆転を正当化しない。

---

### 7.7 既存アーキテクチャ論との関係

CPA は既存のアーキテクチャ論を否定しない。CPA は、それらの理論が「論理的構成」においてのみ適切に機能していた理由を説明し、「存在論的構成」においてもなお一貫した原理を適用するための拡張理論として位置づけられる。

概念の純粋性を原則として強調する既存のアーキテクチャ論は、論理的構成においては完全に正当である——実行基盤が配送装置型である環境では、Core は純粋に保たれるべきであり、保てる。

しかしこれらの理論は、概念の純粋性が原理的に達成できない存在論的構成に対して、十分な処方を提示してこなかった。「できれば純粋にすべき」という規範は、存在基盤型の環境において設計者を無用に苦しめるか、現実との折り合いをアドホックな妥協として処理するかという二択を迫る。

CPA は侵食を定式化することで、この問題を理論内部で解決可能な問題として再記述する。存在論的構成において Core Model が実行世界の存在様式を内包することは、発見可能性公理の帰結として正当化される。設計上の妥協ではなく、構造的必然として理解される。
